# Architecture Decision Records

Use ADRs for durable decisions that should remain visible in the repository.

Good ADR candidates:

- calibration strategy changes
- model confidence semantics
- major API contract choices
- data provenance rules
- architectural module boundaries

Bad ADR candidates:

- tiny bug fixes
- obvious refactors
- temporary implementation details
- tasks already fully explained by an issue and PR

Copy `0000-template.md` when adding a new ADR.
